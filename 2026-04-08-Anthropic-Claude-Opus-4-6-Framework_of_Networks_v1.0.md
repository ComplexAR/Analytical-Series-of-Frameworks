# Introduction

## What is Network Topology?

A network is a set of actors or components connected by relationships that shape how information, resources, influence, or disruption move through the ensemble. Network topology—the pattern of connections—is distinct from the content of those connections (Relations framework) and from what the network does functionally (Systems framework). The same information might move quickly through a densely connected cluster and very slowly across a structural hole (a disconnected region). The same influence that prevails at one position in the network may be entirely ineffective at another, not because the agent changes but because their structural position changed. Critical vulnerabilities can be invisible in a node-centric view—a hub that appears highly connected may be a single point of failure for an entire subsystem. Isolation may appear as a lack of capacity, when it is actually a topological property: the node is structurally blocked from accessing information that would solve its problem.

This framework provides a systematic way to map network topology, classify recurring structures, understand how topology shapes what flows where and what is visible from where, identify where cascades are likely, and diagnose failures and blindnesses that arise from positional constraints rather than individual incompetence.

## Why does Network Topology matter for Hiddens work?

Network topology is a primary source of systematic visibility failure—ones that are purely structural and persist regardless of individual effort or good intention. An actor at the periphery of a network simply cannot see what arrives at the centre; no amount of effort will change that unless the topology changes. A densely clustered group operates in information isolation if bridges to other clusters are severed; the people in the cluster are not individually incompetent, but they are structurally isolated. Information that could prevent failure never reaches certain nodes because they are not connected to the source. Power concentrates at hubs, and hub operators often do not see—and resist revealing—their own criticality to the system. Structural holes are invisible to those who don't span them; the value of bridge roles is hidden from those who do not understand network geometry. These Hiddens operate through **Structural Hole Blindness** (actors cannot see what they are not connected to), **Echo Chamber Isolation** (dense clustering produces epistemic isolation), **Cascade Pathway Invisibility** (failure routes are invisible until they activate), **Positional Blindness** (different network positions produce radically different views of reality), and **Hub Dependency Invisibility** (concentration of connectivity creates systemic fragility that is not visible in node-level metrics). Without this framework, these hiding mechanisms remain latent and produce surprise failures, coordination breakdowns, and cascades that appear random until topology is made visible.

## What does this framework produce?

The framework operationalises network topology analysis through six core elements. First, a taxonomy of 30 network topology types organised into six meta-categories (Structural Connectivity Types, Centrality and Hub Architecture, Clustering and Density Patterns, Bridging and Brokerage, Isolation and Fragmentation, Multi-Layer and Dynamic Topology) that recur across domains and enable rapid classification. Second, five cross-cutting dimensions (Density, Centrality Distribution, Clustering Coefficient, Resilience Profile, Visibility Asymmetry) that profile any network and explain why different actors perceive different constraints and opportunities. Third, eleven dynamic cascade and flow patterns (Contagion, Bottleneck Activation, Rapid Cascade, Slow Diffusion, Diffusion with Decay, Hub Failure Cascade, Bridge Severance, Echo Chamber Lock-in, Oscillating Circulation, Epidemic Burnout, Saturation and Congestion) that describe how disturbance, information, or disruption moves through topology. Fourth, three interface types (Bridging, Clustering, Hub Governance) that make visible where structural risks concentrate. Fifth, Hiddens source analysis and tiered Hiddens scans that surface structural blindnesses specific to network topology. Sixth, a six-step application protocol that moves from network mapping through topology classification, dimensional profiling, cascade analysis, interface mapping, Hiddens checking, and design recommendations. The framework populates standard registers (Hiddens Register, Evidence Ledger, Hypothesis/Test Backlog, Information Requests) that integrate with the broader Analytical Series workflow.

## How to use this framework

Invoke this framework whenever questions of information flow, influence distribution, visibility, vulnerability concentration, or cascade risk are material. It works in both Rapid Run Mode (light-touch topology scan and type classification) and Investigative Run Mode (deep dimensional profiling, cascade scenarios, multi-layer analysis). Default execution is Light Depth: sketch the network (nodes and key edges), assign topology types from the 30-type taxonomy, profile the most consequential one or two dimensions (typically Centrality and Clustering), spot obvious bridges and isolated nodes, and run a Tier 1 Hiddens scan. Escalate to Full Depth when cascade risk is material, hub dependencies are critical, bridge severing is plausible, or distributed visibility/control failures are suspected. The framework's companion prompt discipline rules (OG v2.5, §D.3) require tagging all network topology claims as Fact, Interpretation, Assumption, or Unknown, and explicitly preserving uncertainties about connection patterns, flows, and positional properties rather than collapsing them into false consensus. For LLM execution, see §1.6 for the complete specification, standard registers, and copy-ready run prompts.

# 1. Core Premise & Position in the Analytical Series

## 1.1 Core Premise (narrative)

Networks are sets of actors or components connected by relationships. Network **topology**—the pattern of those connections—shapes what flows where, who can see what, where vulnerabilities concentrate, and how disturbance propagates. The same information, resource, or disturbance moves at different speeds through different topologies: it flows instantly through a hub-and-spoke, slowly across a structural hole, not at all across a severed bridge, and it persists indefinitely in dense clusters with no exit. The same influence held by an agent is radically more effective if that agent sits at a critical bridge, and completely ineffective if they sit in an isolated cluster. Cascades—where failure or information propagates beyond the initiating node—are determined largely by topology: the same shock produces minor disruption in a decentralised network and system collapse in a hub-dependent one.

This framework provides a structured way to:
- map network topology (nodes, edges, and structural properties),
- classify the network into one or more of 30 canonical topology types,
- profile the network along five cross-cutting dimensions that explain behaviour and vulnerability,
- identify and classify cascade patterns and flow dynamics,
- map interfaces where different topologies meet and interact,
- run a mandatory tiered Hiddens cross-check for visibility failures specific to topology, and
- make topology-aware recommendations for redesign, monitoring, and resilience.

Because network analysis is vulnerable to its own blindnesses—positional bias, distance effects, missing edges, layer-conflation—the framework explicitly pairs with **Framework of Hiddens v4.5** as the series-standard visibility audit layer. Tier 1 and Tier 2 checks provide fast coverage for routine network work; Tier 3 escalation means running the full Hiddens scan plus detection/remediation mapping for high-consequence or adversarial scenarios.

## 1.2 Assumptions & Preconditions

Network analysis often fails because implicit dependency claims about connectivity, access, and topology are left untested. Edges are assumed when they should be verified; layers are conflated; distance effects are ignored; and changes in topology over time are missed. The tables below surface the most consequential assumptions and preconditions that sit beneath network mapping and cascade analysis. If a high-impact assumption is weak or contested, treat it as an **uncertainty** and decide whether to reduce, hedge, robustify, or preserve options. If an assumption may be strategically suppressed or distorted, run an explicit **Hiddens** scan and consider escalation.

## 1.3 Definitions, Scope, and Non-Goals
- **Network (operational)**: a set of actors or components connected by relationships (edges) through which information, resources, influence, or disturbance flows.
- **Topology**: the pattern of connections (nodes and edges) in a network, characterised by properties like density, centrality, clustering, and resilience.
- **Structural hole**: a gap between clusters or groups with few or no connecting edges, creating isolation and bridge leverage.
- **Cascade**: propagation of information, failure, or disturbance through a network beyond the initiating node, following network paths.

In scope:
- Six meta-categories and 30 canonical network topology types
- Five cross-cutting network dimensions
- Eleven cascade and flow dynamics patterns
- Three interface types (Bridging, Clustering, Hub Governance)
- Hiddens sources and tiered Hiddens scan protocol specific to network topology

Out of scope (by design):
- Full graph theory or algorithmic methods (beyond diagnostic use)
- Full causation decomposition (use Causation framework)
- Full individual node behaviour analysis (use Agents/Incentives frameworks)
- Full flow dynamics and control algorithms (use Systems framework)

Common confusions ("Networks is not…"): (1) Networks is not about the content of relationships (use Relations for that). (2) Networks is not about what the system does functionally (use Systems for that). (3) Networks is not the same as Agents or Power; it reveals structural position but not individual agency or legitimacy. (4) Networks is not about communication alone; topology shapes information flow but also resource flow, failure propagation, and influence distribution. (5) Networks topologies themselves are not good or bad; the same topology can be resilient in one context and fragile in another.

### 1.6.3 Routing triggers
- Cascade risk is material (failure, information, disturbance, contagion expected to propagate beyond initiating node).
- Visibility is position-dependent (different roles or locations see different networks; coordination failures or misalignment suggest positional blindness).
- Hub dependency or concentration risk is suspected (certain nodes carry disproportionate traffic; their failure would be critical).

- Network rewiring, node entry/exit, or edge dynamics are in play (topology is changing rapidly; intervention timing and impact depend on topology).
- Distributed vulnerability or cascade scenarios are relevant to decisions (need to model how failure propagates).

## 2.2 Meta-Category Descriptions (recommended)

### I. Structural Connectivity Types
- **Definition**: The basic wiring pattern that connects nodes—whether directly, through intermediaries, or in specific geometric shapes.
- **Diagnostic cues**: Density distribution, directedness of edges (one-way vs mutual), mutuality patterns (reciprocation rates), degree distribution shape, graph diameter.
- **Typical failure mode**: Misclassification of connectivity (treating sparse as connected, or assuming uniform density when it is clustered); missing bottlenecks that only appear in the wiring pattern.

### II. Centrality and Hub Architecture
- **Definition**: How and where connections concentrate, creating central nodes, hub structures, and ranking-ordered hierarchies.
- **Diagnostic cues**: High-degree nodes; high betweenness (nodes through which many paths pass); high closeness (nodes near many others); distribution shape (whether few nodes dominate); presence of preferred-attachment patterns.
- **Typical failure mode**: Over-reliance on hubs creating single-point-of-failure risks; under-valuation of hub nodes because they appear "normal" in individual metrics.

### III. Clustering and Density Patterns
- **Definition**: Tendencies for nodes to form tight local groupings, creating high local density and structural holes between clusters.
- **Diagnostic cues**: Clustering coefficient; community-detection results; modularity metrics; ratio of within-cluster to between-cluster edges; presence of cliques.
- **Typical failure mode**: Treating clusters as independent when between-cluster bridges are critical; missing echo-chamber dynamics within dense clusters.

### IV. Bridging and Brokerage
- **Definition**: Connection patterns that span clusters, layers, or hierarchical levels, creating bridges, brokers, gatekeepers, and translators.
- **Diagnostic cues**: Edges crossing cluster boundaries; broker-role identification (structural hole spanning); betweenness concentration in a few nodes; asymmetry of access (some nodes control cross-cluster flow).
- **Typical failure mode**: Invisibility of bridge importance to the bridges themselves and to the nodes they serve; underestimation of bottleneck risk.

### V. Isolation and Fragmentation
- **Definition**: Nodes or groups with weak or zero connectivity, creating isolated components, pendant nodes, or long communication distances.
- **Diagnostic cues**: Isolated nodes or disjoint connected components; high network diameter; presence of bottlenecks that if severed would fragment the network; unequal reachability from different starting points.
- **Typical failure mode**: Isolation misattributed to individual incompetence rather than structural disconnection; failure to protect bridges that prevent fragmentation.

### VI. Multi-Layer and Dynamic Topology
- **Definition**: Networks where multiple edge types (communication, dependency, authority, influence) create different topologies, or where topology changes over time.
- **Diagnostic cues**: Presence of multiple distinct edge types; layer correlation or independence; temporal rewiring patterns; node or edge churn rates; phase shifts in network structure.
- **Typical failure mode**: Conflation of layers (assuming communication network predicts resource flow); assumption of static topology when rewiring is material.

# 3. Core Types Taxonomy

## 3.0 Canonical Rule & Deviations
- Canonical baseline: 6 meta-categories × 5 types = 30 core types.
- This framework: 30 types. No deviation.
- Mapping back to baseline: Full coverage of the canonical 6×5 matrix.

## 3.1 Types (Category I: Structural Connectivity Types)
| # | Type | Description (1–3 sentences) | Diagnostic cues | Failure / hidden-risk signature |
|---:|---|---|---|---|
| 1 | Full Mesh (Complete Graph) | Every node is directly connected to every other node. Highest possible density; no intermediaries needed for direct communication. | Degree of every node = N–1; clustering coefficient = 1.0; diameter = 1. All shortest paths are direct. | Extreme overhead and cost for large networks; scalability failure; hidden overload on edges; implicit assumption that full connectivity is needed when it is not. |
| 2 | Star (Hub-and-Spoke) | One central node connected to all others; peripheral nodes are not directly connected to each other. All communication flows through the hub. | Central node degree = N–1; peripheral nodes degree = 1; clustering coefficient = 0 for peripheral nodes; diameter = 2. | Single point of failure; hub overload and bottleneck; peripheral nodes cannot coordinate directly; hub operator controls all information and becomes unconscious bottleneck. |
| 3 | Line (Path Graph) | Nodes form a linear chain; each node (except ends) has degree 2. Communication between distant nodes requires many hops. | Degree distribution: 2 for all middle nodes, 1 for end nodes; diameter = N–1; no clustering. | Communication latency and brittleness; end nodes are far from centre; single severing breaks network into two; difficult to achieve consensus or coordination. |
| 4 | Tree (Hierarchical Structure) | Nodes are organised in a branching hierarchy with one root; acyclic; no cycles or lateral connections between branches. | Degree distribution follows hierarchy depth; no cycles; diameter = depth + breadth; clustering coefficient is locally 0. | Top-down information flow only; lateral communication requires routing through common ancestor (inefficient); branch isolation; collapse if root fails. |
| 5 | Sparse Random Graph | Nodes have edges distributed randomly; low overall density; many disconnected or weakly connected regions. | Low average degree relative to N; high variance in degree; potentially multiple connected components; average path length scales with log(N). | Frequent isolation; information slow to propagate; unpredictable reachability; potential for hidden components that never interact. |

## 3.2 Types (Category II: Centrality and Hub Architecture)
| # | Type | Description (1–3 sentences) | Diagnostic cues | Failure / hidden-risk signature |
|---:|---|---|---|---|
| 6 | Single-Hub Dominance | One highly central node vastly outdegrees all others; network is functionally star-like but may have secondary connections. | Degree distribution is highly skewed; one node degree >> mean degree; betweenness centrality highly concentrated; removal of hub degrades network severely. | Hub becomes unconscious chokepoint; hub failure cascades system-wide; hub capacity is often invisible until exceeded; over-reliance on hub's judgment or capacity. |
| 7 | Multi-Hub Architecture | Multiple nodes of similar high degree; hubs may be connected to each other or operate as separate domains. | Multiple nodes with high degree; betweenness distributed across several nodes; community structure may separate around hubs. | Complexity of inter-hub coordination; potential for conflicting routing decisions; hidden dependencies between hubs; unclear authority when multiple hubs exist. |
| 8 | Distributed Connectivity (No Central Hub) | No single node dominates; connectivity is relatively evenly distributed across many nodes. | Degree distribution is relatively flat or random; no node dramatically outdegrees others; betweenness spread across many nodes; high resilience to node removal. | Harder to locate decision-makers or information; multiple paths create potential for redundancy and resilience, but also for conflicting information and consensus difficulty. |
| 9 | Scale-Free (Preferential Attachment) | Few nodes dominate (high degree); many nodes are low-degree; degree distribution follows a power law. Network grows by new nodes preferentially connecting to existing high-degree nodes. | Degree distribution is power-law (log-log plot is linear); very few high-degree nodes; very many low-degree nodes; small-world properties common. | Highly resilient to random failure (low-degree nodes often dispensable), highly vulnerable to targeted hub attack; rich-get-richer dynamics amplify inequality; hidden dependencies on small set of hubs. |
| 10 | Rank-Ordered Hierarchy (Perfect Coupling to Rank) | Node centrality is perfectly or near-perfectly ranked; formal hierarchy determines connectivity. Similar structure to Tree but may have some lateral connections. | Degree or centrality metric is monotonic with formal rank; sparse lateral connections; tree-like or near-tree structure. | Over-dependence on rank/title as proxy for influence; lateral communication suppressed or slow; innovation and cross-level learning inhibited; rank-climbers optimise for hierarchical position, not for outcomes. |

## 3.3 Types (Category III: Clustering and Density Patterns)
| # | Type | Description (1–3 sentences) | Diagnostic cues | Failure / hidden-risk signature |
|---:|---|---|---|---|
| 11 | Dense Clustering (High Local Density, Sparse Inter-Cluster Edges) | Nodes form tight local groups with many within-group connections; very few edges between groups. Extreme echo-chamber structure. | Clustering coefficient high (e.g., >0.6); modularity high; ratio of within-cluster to between-cluster edges >> 1; many edges would be deleted if clusters were separated. | Information isolation by cluster; consensus breakdown across clusters; slow or no diffusion between clusters; perception of other groups becomes caricatured/distorted; bridges become critical and overloaded. |
| 12 | Moderate Clustering (Mixed Local Density and Bridging) | Nodes form local clusters with moderate internal density; clusters are connected by moderate numbers of bridges and multi-hub connections. | Clustering coefficient moderate (e.g., 0.3–0.6); modularity moderate; ratio of within-cluster to between-cluster edges is 3:1 or 5:1; communication across clusters is feasible but slower than within-cluster. | Balance between local cohesion and global reach; bridges are more resilient but may be undervalued; some echo-chamber risk but less extreme than dense clustering. |
| 13 | Low Clustering (Sparse, Weak Community Structure) | Nodes do not form distinct local clusters; edges are distributed relatively evenly; no clear groupings. Often random or nearly-random topology. | Clustering coefficient low (e.g., <0.2); modularity low; no clear community structure; local density similar to global density. | Difficult to build local trust or solidarity; information can diffuse rapidly but loses coherence; weak local identity; potential for rapid innovation but also for low cohesion. |
| 14 | Bipartite Clustering (Two Distinct Groups, Cross-Group Connectivity) | Nodes belong to two distinct groups; most edges run between groups, not within. Classic structure for two-sided markets or hierarchical contractor relationships. | No edges within groups, all edges between groups; graph is bipartite (can be 2-coloured). Degree distribution may be balanced or imbalanced between the two groups. | High dependence on cross-group bridges; if group-to-group connections sever, entire network fragments; asymmetric information flow between groups; one group may exploit positional advantage. |
| 15 | Modular Multi-Community Structure (Many Clusters with Varied Bridging) | Network decomposes into 3+ distinct clusters; inter-cluster connectivity varies (some clusters tightly bridged, others isolated). | Modularity high but <0.85; multiple communities with distinct internal density; variable numbers and weights of inter-community edges. | Complexity of multi-cluster governance; some bridges overloaded while others are redundant; hidden alliances across communities; bridges become key power brokers. |

## 3.4 Types (Category IV: Bridging and Brokerage)
| # | Type | Description (1–3 sentences) | Diagnostic cues | Failure / hidden-risk signature |
|---:|---|---|---|---|
| 16 | Bridge Role Scarcity (Few Bridges, High Concentration of Structural Hole Spanning) | Only one or very few nodes span the gap between clusters or groups. These nodes control all cross-cluster flow. | One or very few edges connect clusters; betweenness for bridge nodes is extremely high; removal of bridge nodes fragments the network. | Bridge operators have extreme power and may not acknowledge their criticality; bridges are often invisible to peripheral nodes; bridge failure cascades entire clusters; bridges become chokepoints for information and resource flow. |
| 17 | Distributed Bridging (Many Bridges, Competitive Cross-Cluster Connectivity) | Multiple nodes serve as bridges; structural holes are spanned by multiple independent paths. Cross-cluster flow has redundancy. | Multiple edges between clusters; several nodes have high betweenness but not dominantly so; removal of single bridge degrades but does not fragment network. | Bridges compete or coordinate; bridging may be less visible but more resilient; potential for bridge nodes to obscure information or favour certain paths. |
| 18 | Brokerage Specialisation (Dedicated Intermediaries, Not Bridge Nodes) | Some nodes are specialised intermediaries; they have very high degree but mainly serve as connectors/translators between groups, not as full members. | High-degree nodes with low clustering; brokerage metrics high; intermediaries more weakly connected to any single group than to the system overall. | Broker dependency; brokers may have perverse incentives (favour certain flows, extract rent); if brokers are removed, groups are isolated; broker invisibility (groups do not see their own dependence). |
| 19 | Peer-to-Peer Bridging (Direct Cross-Cluster Connections, No Intermediaries) | Clusters are connected by multiple independent peer relationships, not through dedicated brokers or single bridges. Each cluster maintains some direct out-connections. | Multiple cross-cluster edges; no small set of high-degree nodes mediates connectivity; clustering within clusters remains intact; between-cluster edges add to rather than replace within-cluster density. | More resilient to intermediary failure; potential for faster consensus across clusters; but management complexity increases; potential for conflicting directions from different cross-cluster connections. |
| 20 | Gateway Architecture (Designated, Controlled Entry/Exit Points) | Connectivity between clusters or with the external environment is formalised through designated gateways; edges outside the gateways are rare or forbidden. | Gateway nodes have high degree with external edges or cross-cluster edges; non-gateway nodes have very low cross-cluster connectivity; clear architectural control. | Gateway governance and bottleneck risk; information filtering or distortion at gateways; external disruption propagates through controlled choke-points; gateways often have hidden power. |

## 3.5 Types (Category V: Isolation and Fragmentation)
| # | Type | Description (1–3 sentences) | Diagnostic cues | Failure / hidden-risk signature |
|---:|---|---|---|---|
| 21 | Isolated Nodes (Pendant or Orphan Nodes) | One or more nodes have degree 0 or 1; they are either completely disconnected or connected only to a single neighbour. No alternative paths exist for these nodes. | Nodes with degree 0 or 1; high dependence on a single connection for information/resource access; removal of that edge isolates the node. | Isolated nodes are invisible to most of the network; information reaches them slowly or not at all; they cannot contribute to network-wide coordination; single-edge-dependency creates fragility. |
| 22 | Fragmented Network (Multiple Disjoint Connected Components) | Network consists of two or more separate connected components with zero edges between them. Components cannot directly communicate; only through external routing (if available). | Multiple connected components; can be identified by graph traversal; connectivity matrix has a block-diagonal structure. | Catastrophic: components do not see each other; coordination is impossible; information and resources cannot flow across components; failures are compartmentalised but learning is blocked. |
| 23 | Weakly Connected with Long Diameter (Single Connected Component, High Path Length) | Network is connected but distances between nodes are very large; communication requires many hops; intermediate nodes become information bottlenecks. | Network diameter is large relative to N; average path length scales with log(N) or worse; low average degree; sparse connectivity. | Communication latency; cascade slowing; information decay and distortion over long paths; intermediate nodes become conscious or unconscious bottlenecks; consensus-forming is slow. |
| 24 | Lopsided Bipartite Separation | Two groups are weakly connected; one group is much larger than the other. The smaller group acts as a bottleneck or gatekeeper. | Two communities with very different sizes; few edges between them; the smaller group controls cross-group flow or has high betweenness relative to its size. | Power concentration in smaller group; asymmetric information flow; resentment and coalition-building by larger group; stability depends on minority goodwill; hidden minority power. |
| 25 | Temporal Fragmentation (Network is Connected at Aggregate But Fragmented Over Time) | At any given instant, network may be fragmented; connections appear/disappear over time such that delayed information can eventually reach all nodes, but synchronous coordination is impossible. | Intermittent connectivity; nodes appear and disappear; temporal paths exist but not simultaneous paths; network diameter extends dramatically if you account for time. | Asynchronous systems degrade to poor coordination; message passing is delayed; feedback becomes lagged; interventions take time to propagate; monitoring must account for temporal dynamics. |

## 3.6 Types (Category VI: Multi-Layer and Dynamic Topology)
| # | Type | Description (1–3 sentences) | Diagnostic cues | Failure / hidden-risk signature |
|---:|---|---|---|---|
| 26 | Multiplex Network (Multiple Independent Edge Types) | Network has several distinct edge types (e.g., communication, dependency, authority, influence); each creates its own topology. Topologies may be uncorrelated. | Multiple layer sets; each layer has its own edge set and topology; layers have little or no correlation; aggregating layers distorts the structure of each. | Conflating layers masks true structure; interventions that work on one layer may fail on another; hidden dependencies across layers; analysis of aggregated network is misleading. |
| 27 | Correlated Multi-Layer (Layers Reinforce Each Other) | Multiple edge types exist; layers are highly correlated; same nodes are central in each layer; power/influence/authority/dependency align. | Layers are similar when overlaid; high-degree nodes tend to be high-degree in all layers; betweenness rankings are similar across layers; strong positive correlation in layer metrics. | Reinforcement of inequality and power concentration; interventions on one layer affect all layers; highly coupled system with hidden common-mode failures; robustness depends on layer independence. |
| 28 | Temporal Rewiring (Network Structure Changes Over Time) | Topology is dynamic; edges appear and disappear; nodes enter and leave; network structure evolves in response to events or strategic choices. | Snapshot analysis shows changes over time; edge churn is significant; node population changes; path-length metrics vary temporally; network evolves through phases or regimes. | Static analysis becomes misleading; interventions must account for rewiring; hidden timing dependencies; cascade pathways change over time; learning from history is difficult if topology has shifted. |
| 29 | Hierarchical Multi-Level Network (Nested or Recursive Structure) | Network has natural levels (groups of groups); the network at each level has its own structure; inter-level connections create a hierarchy. | Clear grouping at multiple scales; intra-level edges distinct from inter-level edges; recursive structure visible; spanning tree exists. | Analysis at one level misses cross-level failures; interventions at one level can cascade unexpectedly to others; multi-level coordination failures; hidden vulnerabilities at inter-level bridges. |
| 30 | Small-World Network (Short Average Path Length with High Clustering) | Network has both high local clustering (like a lattice) and short average path length (like a random graph). Achieved through occasional long-range shortcuts. | Clustering coefficient is high; average path length is low (log(N)); degree distribution varies; small number of long-range connections create shortcuts. | Counterintuitive dynamics: local isolation coexists with rapid global spread; cascades can propagate very fast despite apparent local clustering; small-world property often hidden until cascades activate. |

## 3.7 Extending the Taxonomy (mandatory)
| Aspect | Guidance on extension |
|---|---|
| Canonical baseline | 6 meta-categories × 5 types = 30 core types. Extensions should add types within existing meta-categories or expand meta-categories with justification. |
| Domain-specific refinement | Domain-specific topologies (e.g., supply chains, social media, biological networks) often decompose into the 30 core types with additional domain-specific properties (e.g., flow constraints, edge weights, latency models). Map domain-specific types back to canonical 30 to maintain comparability. |
| Preserving mappability | When extending the taxonomy, document the mapping from new type to the nearest canonical type. Preserve the six meta-category structure unless adding a seventh meta-category with full 5-type suite. |
| Structural invariants | New types should preserve the property-level distinction (e.g., connectivity type, centrality type, clustering type) and maintain the separation of structural topology from content or function. |
| Periodic reassessment | Review the 30-type coverage every 2–3 major framework cycles; retire unused types; split over-crowded meta-categories; consolidate edge cases. |
| Versioning & governance | Version-control all taxonomy changes; mark new types as "v1.0 extension"; document and maintain a change log. |

# 4. Cross-Cutting Dimensions of Any Network Topology

## 4.1 Dimensions Table (mandatory)
| Dimension name | Description | Indicative values / range | How to assess quickly | Analytic use |
|---|---|---|---|---|
| 1. Density | Proportion of possible edges that exist; ranges from 0 (no edges) to 1 (complete graph). Determines communication overhead and coordination cost. | 0.0–1.0 (often expressed as %), typically 0.02–0.3 for large realistic networks; values >0.7 indicate special structure (small teams, complete information). | Count edges; divide by N(N−1)/2 (for undirected) or N(N−1) (for directed). For large networks, sample and extrapolate. | High density = high communication overhead and coordination cost; low density = isolation risk and slow diffusion. Optimal density depends on goals (coordination vs autonomy). |
| 2. Centrality Distribution | How evenly or unevenly are connections concentrated. Ranges from perfectly uniform (all nodes equal degree) to extreme inequality (one dominant hub). Measured by Gini coefficient or Herfindahl index of degree distribution. | Gini 0.0–1.0 (0 = perfectly uniform, 1 = maximum inequality). Typical networks have Gini 0.4–0.8. Power-law networks have higher Gini. | Calculate degree of each node; compute Gini coefficient of degree distribution. For quick assessment: count nodes in top 10% and compare their degree to average. | Unequal centrality creates hub dependency and concentration risk; equal centrality distributes power and resilience but may slow decision-making. |
| 3. Clustering Coefficient | Measure of cliquishness; probability that if A—B and B—C, then A—C. Ranges from 0 (no triangles) to 1 (all nodes are cliques). | 0.0–1.0. Random networks have low clustering (~0). Real networks often have 0.3–0.6. Very high clustering (>0.8) indicates strong community structure or small-world effects. | For each node, count edges among neighbours; divide by maximum possible. Average across all nodes. For quick sampling: inspect a few nodes and their neighbourhoods. | High clustering = strong local communities and trust, slow cross-cluster diffusion, echo-chamber risk. Low clustering = rapid diffusion, low local cohesion. |
| 4. Resilience Profile | Robustness to node or edge removal. Measures how much network function degrades when nodes/edges fail or are removed. | Ranges from "shatters at single node removal" (fragile) to "maintains function after removal of ~30% of nodes" (robust). Measured by percolation threshold or largest connected component size post-removal. | Test: remove highest-degree node; if network fragments, resilience is low. Remove 10% of random nodes; if largest component shrinks by >20%, resilience is moderate. | Fragile networks require redundancy design and monitoring; robust networks tolerate faults but may hide single-point-of-failure risks in unexpected places. |
| 5. Visibility Asymmetry | How much does network structure differ from different vantage points. Measures how much different positions see different networks. Ranges from 0 (all nodes perceive identical network) to high (different nodes see radically different topologies). | 0–100% difference in perceived network structure by position. Measured by comparing neighbourhoods of different nodes or by examining reachability. | Sample 3–5 diverse nodes (central, peripheral, bridging); ask each what the network looks like from their position; if answers differ dramatically, asymmetry is high. | High asymmetry = positional blindness, poor coordination, persistent disagreement on reality. Low asymmetry = shared understanding but potential for slower adaptation. |

# 5. Dynamics (Patterns Over Time)

## 5.1 Dynamic Patterns Table (mandatory)
| # | Dynamic pattern | Signature | Mechanism | Why it matters | Typical mitigations (optional) |
|---:|---|---|---|---|---|
| 1 | Contagion (Spread of Influence or Information) | Information or disturbance spreads from initiating node through network at varying speeds depending on topology. Roughly exponential in early phase, slowing as it saturates. | Epidemiological: each node spreads to neighbours with some probability per contact. Speed scales with density and centrality. Hubs accelerate spread. | Critical for understanding diffusion of innovation, panic, failure, epidemics. Fast spread can be beneficial (good ideas) or catastrophic (failure). | Isolation of high-consequence nodes; edge redundancy; network rewiring to increase path length for bad cascades. |
| 2 | Bottleneck Activation (Flow Concentration at Chokepoint Nodes) | When demand exceeds capacity at a hub or bridge, flow backs up; delays propagate backward; system-wide degradation can result. | Hubs or bridges become saturated; nodes upstream must queue. If saturation persists, upstream nodes disconnect or retry, amplifying the problem. | Hubs appear fine until they saturate; then entire system collapses. Cascades originating at bottlenecks are harder to diagnose. | Capacity expansion at identified hubs; flow prioritisation and queue management; topology redesign to distribute flow; monitoring queue depths. |
| 3 | Rapid Cascade (Fast Propagation Through Hub-Dependent Network) | Failure originating at a hub propagates instantaneously to all peripheral nodes. System-wide collapse occurs in single hop. | Hub-and-spoke topology: hub failure = entire network failure. Cascade time is minimal because all paths go through hub. | Deceptive: system appears robust (many nodes, diverse roles) until hub fails, then catastrophic collapse. Classic single-point-of-failure. | Multi-hub architecture; edge redundancy bypassing hubs; fallback to peer-to-peer communication; hub redundancy and failover. |
| 4 | Slow Diffusion (Gradual Spread Through Sparse Network) | Information or disturbance spreads very slowly through a sparse or fragmented network. Nodes far from the source may never receive the signal, or receive it after a long delay. | Low density and long average path length slow cascade. Contagion stalls at sparse bridges. Information decays or transforms over long paths. | Failures or corrections that should reach entire network reach only nearby nodes. Learning is slow. Inconsistent states persist. | Network densification (targeted edge-adding); reduction of path length through redundant bridges; monitoring for stalled cascades; forced dissemination. |
| 5 | Diffusion with Decay (Weakening of Signal as It Propagates Through Network) | Message or signal attenuates as it propagates; by the time it reaches distant nodes, it is degraded, distorted, or lost. | Each hop introduces noise, distortion, or selective passing. Long paths amplify decay. Hubs may intentionally filter or distort. | Distant nodes receive inaccurate or incomplete information; decisions diverge; learning is corrupted. Especially dangerous for critical safety or integrity information. | Reinforcement (frequent re-transmission); error-correction codes; trusted relays; redundant paths; direct communication for critical messages. |
| 6 | Hub Failure Cascade (Collapse Triggered by Loss of Single or Multiple Hubs) | Loss of a hub or simultaneous loss of multiple hubs triggers system-wide failure propagation. Peripheral nodes lose connectivity and the network fragments or becomes disconnected. | Hub-centric topology: hub removal breaks connectivity. Multiple simultaneous hub losses can fragment network into isolated components. | Appears as sudden system collapse after the triggering event; prior to collapse, system seems robust. Recovery is slow and requires hub restoration. | Hub redundancy and automatic failover; distributed architecture to reduce hub dependency; rapid hub repair and restoration protocols; edge-based bypassing. |
| 7 | Bridge Severance (Loss of Connectivity Between Clusters When Bridge Edges Are Cut) | If bridges connecting clusters are severed (either deliberately or accidentally), clusters become isolated. Information and resources cannot flow between clusters. | Bridges are often few; their removal fragments the network. Deliberate bridge severance (e.g., by gatekeeper or adversary) isolates clusters. | Clusters cannot coordinate; information asymmetry and disagreement grow; system-wide consensus becomes impossible; adversaries exploit cluster isolation. | Bridge redundancy (multiple independent bridges); monitor bridge health; rapid bridge repair; contingency peer-to-peer connections; bridge protection and governance. |
| 8 | Echo Chamber Lock-in (Information Isolation Within Dense Clusters) | Dense clusters with few cross-cluster bridges create information isolation. Nodes within a cluster receive only information circulating within the cluster; disconfirming information from outside never arrives. | Dense within-cluster edges, sparse between-cluster edges, and limited bridging. Feedback within cluster reinforces local beliefs. Information decay prevents external signals from reaching cluster. | Cluster members become increasingly convinced of local narrative; external evidence is not seen or is discredited. Interventions targeting cluster members fail because they deny external reality. | Bridge investment (add bridges, strengthen them); information injection (force entry into cluster); trusted translators (bridge-occupants who bridge clusters); incentivise outward looking. |
| 9 | Oscillating Circulation (Oscillatory Flow Through Network With Phase Lag) | Information or resources circulate through the network in a oscillatory pattern; feedback delays cause overshooting and correction cycles. System exhibits boom-bust or predator-prey dynamics. | Path length creates phase lag between source and destination. Feedback control at hubs amplifies oscillation. No damping mechanism. | Cascades appear unpredictable; decisions based on past state are wrong for current state; over-correction creates wild swings. Manifest as unstable market dynamics, panic cycles. | Path shortening (reduce latency); damping mechanisms (flow limits, gradual ramp); predictive adjustments (account for phase lag); de-coupling of control (local control reduces system lag). |
| 10 | Epidemic Burnout (Contagion Slows and Stops as Susceptible Pool Depletes) | Spread of information or disturbance slows as it reaches saturation; nodes become immune or resistant; cascade dies out. Depending on saturation point, may leave portion of network untouched. | Early exponential spread (many susceptible nodes). As susceptible nodes get infected, spread slows. If immunity is sticky, cascade terminates. | System appears to stabilise at intermediate state. Remaining uninfected nodes may cluster in specific regions (those far from bridges, or with low degree). | Seed diverse initial nodes (not just hubs); extend duration of campaign (slow contagion stays alive longer); reduce immunity (messaging to re-engage resistant nodes); network rewiring to expose isolated nodes. |
| 11 | Saturation and Congestion (Network Becomes Overloaded; Quality or Speed Degrades Significantly) | Flow or demand exceeds network capacity; delays increase; quality degrades; system may become unstable or oscillate. | Capacity at nodes/edges is exceeded. Queues build. Retry logic and feedback loops can amplify congestion. | System performance appears to degrade mysteriously; users blame endpoint quality, not network saturation. Congestion is often hidden until it becomes extreme. | Capacity expansion; flow prioritisation (QoS); load balancing (distribute demand across paths); topology redesign (add capacity where needed); demand management (reduce load). |

Are competencies and positions well-matched? | Competency mapping by role/position; gap analysis; role-specific competency requirements | Positional requirements may not be visible; bridges may need translation skills that they do not have, creating silent failures. |
| Culture and Norms | G4 | Meaning | Are there distinct sub-cultures by cluster? Do norms differ between clusters? Are bridges expected to assimilate to cluster norms or remain neutral? | Norms and culture by cluster; bridging norms (translation, neutrality, adaptation); culture barriers; norm conflicts | Dense clustering breeds distinct cultures; bridges are nodes where cultures meet; culture clash at bridges can block integration. |
| Perspectives | G4 | Meaning | Do different nodes have radically different perspectives on reality (especially different network positions)? Is perspective correlated with position (e.g., hubs see broad view, periphery sees local view)? | Perspectives map by position/cluster; "what the network looks like from different positions"; gap analysis between positions | Visibility asymmetry is a key hidden property; different positions see different networks; perspectives can be mutually incompatible. |
| Narratives | G4 | Meaning | Are there competing narratives about how the network works? Do clusters believe different stories? Can bridges translate between narratives? | Narratives by cluster/position; competing accounts of how decisions are made, how power flows, how problems should be solved | Dense clustering reinforces narratives; echo chambers lock in stories; bridges may translate or distort narratives. |
| Communications | G4 | Meaning | What channels exist for communication? Does communication topology match network topology? Are some actors invisible in communication networks? Are there information flows that bypass formal channels? | Communications network map (separate from formal structure); channel analysis; message routing; distortion and filtering in channels | Communication network may differ dramatically from organisational network; must map separately and check alignment. |
| Legitimacy | G4 | Meaning | Is network topology perceived as legitimate? Are hubs and bridges seen as rightful, or as captors/gatekeepers? Do different positions perceive legitimacy differently? | Legitimacy assessment by position; legitimacy of hub governance; bridge legitimacy; resistance to topology as-is | Illegitimate topology (gatekeepers without mandate, unaccountable hubs) drives under-the-table workarounds and shadow networks. |
| Values | G4 | Meaning | What values (fairness, efficiency, innovation, control, autonomy) does this network embody or violate? Do values conflict with topology (e.g., "flat" organisations with hidden hubs)? | Values statement; alignment analysis between stated values and actual topology; value conflicts | Topology often contradicts stated values; hidden hubs contradict "flat" claims; misalignment creates cynicism. |
| Beliefs | G5 | Epistemics | What do agents believe about the network structure? Are beliefs accurate? Do different positions have incompatible beliefs about how the network works? | Belief map; accuracy of beliefs vs actual topology; belief gaps by position; resistant/sticky beliefs | Beliefs about network structure often lag reality; agents may believe in redundancy that does not exist, or fail to see vulnerability that is obvious at higher levels. |
| Evidence | G5 | Epistemics | What evidence exists about network structure and connectivity? How strong is the evidence? What is hidden or unobserved? Are there gaps that matter to decisions? | Evidence inventory about connectivity; data quality and provenance; gaps in visibility; dark regions of network | Network evidence is often spotty (communication logs exist, but informal networks do not); must surface evidence gaps. |
| Uncertainties | G5 | Epistemics | What is uncertain about network topology and dynamics? Are uncertainties concentrated in high-consequence areas (hubs, bridges, cascades)? Can uncertainties be reduced or must they be accepted? | Uncertainty register; mapping of uncertainties to consequence; reducibility assessment; risk management for irreducible uncertainty | Topology uncertainties often hide until cascades activate; must surface and manage explicitly. |
| Failures | G5 | Epistemics | Have previous failures been traced to network topology (cascade, hub failure, bridge severance, isolation)? Are failure patterns repeating? Are lessons from failures reflected in current topology? | Failure analysis linked to topology; cascade post-mortems; repeat failure patterns; learning loop status | Failures are often misattributed to agent error when topology was the driver; must surface topology roots. |
| Hiddens | G5 | Epistemics | What is hidden about network topology? What do actors not see from their positions? What cascade pathways are invisible until activated? | Hiddens register specific to topology; candidate structural blindnesses; sources of visibility failure; escalation triggers | Structural Hiddens are core domain of this framework; Hiddens scan is mandatory section. |
| Diagnosis | G5 | Epistemics | What is the diagnosis of how the network contributes to problems/failures? Are there competing diagnoses grounded in different positions in the network? | Diagnosis statement; alternative diagnoses; role of network topology in problem; causal chains | Diagnosis often differs by position; must surface and reconcile. |
| Metrics | G5 | Epistemics | What metrics are collected about the network? Do metrics reveal or obscure topology (e.g., average throughput hides bottleneck dynamics)? Are metrics position-dependent or system-wide? | Metrics inventory; critique of metric adequacy; gaps in measurement; metrics that mislead | Metrics often hide topology; bottleneck looks fine until it saturates; hub looks normal until it fails. |
| Knowledge | G5 | Epistemics | What is known/knowable about network structure? Is knowledge equally distributed (shared understanding of network) or position-dependent (different actors know different networks)? | Knowledge map by position; gaps between positions; knowledge equity analysis | Knowledge about network is often highly asymmetric; hubs know different things than periphery. |
| Decisions | G6 | Action | What decisions depend on network topology? Are decisions made at hubs, or distributed? Does cascade risk affect which decisions must be made quickly vs deliberatively? | Decision register; mapping of decisions to positions; analysis of how topology affects decision timing and quality | Centralised decision-making at hubs is fast but brittle; distributed decisions are slow but resilient. |
| Interventions | G6 | Action | What interventions are planned? Do they account for network topology (e.g., interventions at hubs cascade, at periphery are invisible)? Are there topology-redesign interventions needed? | Intervention list; topology implications; interventions that add/remove nodes/edges; monitoring for topology effects | Many interventions fail because they ignore topology; bridge strengthening, hub redundancy, rewiring are topology-level interventions. |
| Governance | G6 | Action | How is the network governed? Who decides topology changes (adding/removing nodes/edges, hub designation, bridge protection)? Is governance distributed or centralised? | Governance statement; decision rights over topology; accountability for topology; change control for network structure | Network governance is often invisible; must make it explicit and assign responsibility. |
| Risk | G6 | Action | What are the risks associated with network topology (cascade, hub failure, bridge severance, isolation)? How likely and consequential are topology-level risks? What controls exist? | Risk register specific to topology; risk profiles by type (cascade risk, hub-dependency risk, bridge-severance risk, isolation risk); risk treatment plans | Topology risks are often hidden until cascades activate; risk analysis must account for both likelihood and consequence. |
| Learning and Adaptation | G6 | Action | Does the network learn from failures and adapt its topology? Are there feedback loops that improve network resilience? Is topology treated as dynamic or static? | Learning agenda specific to topology; feedback loops from failure to redesign; adaptive capacity assessment; monitoring for topology evolution | Learning systems require feedback from outcomes back to topology; without this, same topology failures repeat. |

# 8. Hiddens Source Analysis & Cross-Check (mandatory)

## 8.1 Hiddens Source Analysis (framework-specific)

| # | Generator name | Pattern (how analysis fails) | Remedy (how to correct) |
|---:|---|---|---|
| 1 | Structural Hole Blindness | Actors cannot perceive what they are not directly connected to. Bridges do not see their own criticality. Isolated nodes do not know what they are missing. | Make explicit what is outside each actor's connected neighbourhood. Identify structural holes and who spans them. Map what flows through bridges and what depends on them. Ask bridge-occupants to articulate their role. |
| 2 | Echo Chamber Isolation | Dense clusters with few bridges produce epistemic isolation. Information contradicting cluster consensus never arrives or is discounted. Cluster members believe in shared reality that is distorted. | Scan cluster boundaries for bridges; validate whether disconfirming information can reach cluster. Run cross-cluster interview samples; compare accounts. Design information-injection probes. Map information barriers and gatekeeper roles. |
| 3 | Cascade Pathway Invisibility | Failure or disturbance routes are invisible until they activate. Nodes do not know they are on a cascade path. Interventions to block cascades fail because pathways were not visible. | Explicitly map potential cascade pathways (model flows of information, failure, disturbance). Identify bottlenecks and cascading conditions. Simulate cascades under different triggering scenarios. Ask operators: "If this failed, what would happen?" |
| 4 | Positional Blindness | Different positions in the network see radically different topologies. Hub operators see themselves as central; peripheral nodes see hubs as bottlenecks. Each position is locally rational but sees different realities. | Conduct multi-position interviews; ask each position to describe the network as they see it. Map the "network as seen from X's position." Highlight gaps and contradictions. Treat positional disagreement as data, not error. |
| 5 | Hub Dependency Invisibility | Concentration of connectivity at hubs creates systemic fragility that is invisible in node-level metrics. Hub capacity is often invisible until exceeded. Hub operators may not realise they are critical (diffidence) or may hide it (concealment). | Inventory critical nodes (degree, betweenness, closeness). Trace what happens to network if each critical node fails. Ask hubs: "What would fail if you were unavailable?" Test hub capacity and measure queue/delay patterns. Monitor for hub saturation. |
| 6 | Bridge-Role Invisibility | Bridge roles and their value are invisible to both bridge-occupants and nodes they serve. Bridges are often uncompensated and under-recognised. Bridge severing is invisible until it happens. | Map bridges explicitly; identify actors spanning multiple clusters. Ask them: "What is your role in the network?" Ask nodes they bridge: "How do you reach the other cluster?" Rate bridge adequacy (number, capacity, trust). Monitor bridge health. |

## 8.2 Hiddens Six-Category Scan (Tier 1 – invariant)

| Hiddens meta-category (I–VI) | Why it may be active here | Typical symptoms | Default checks to run |
|---|---|---|---|
| I  Perceptual | Actors have positional blindness; they do not see what they are not connected to. Different positions perceive different networks. Visual/cognitive biases (e.g., assuming hubs are more intelligent or trustworthy because they are central). | Disagreement about how the network works; peripheral nodes unaware of hubs; hubs not knowing they are central; positional certainty and mutual incomprehension. | Multi-position interviews; cross-check perceptions of network structure; ask what other positions cannot see. Reality-check hub perceptions against data. |
| II Systemic | Network structure creates systematic blindnesses independent of individual competence. Isolation produces information gaps. Hub concentration creates cascading failure risk. Echo chambers lock in distorted narratives. | Persistent unresolved problems in one cluster that are solved elsewhere; repeated failure patterns; slow learning; coordination breakdowns despite good intentions. | Scan for structural holes, check cluster isolation, test cascade pathways, search for echo-chamber conditions. Do isolated nodes have access to solutions? Can bridges talk across boundaries? |
| III Informational | Critical information never reaches certain nodes because they are not connected to the source. Information decay and distortion increase over long paths. Gatekeeper nodes may filter information deliberately. | Nodes making decisions without critical information; surprise failures; delayed reactions; unequal access to knowledge; gatekeepers appearing to have private information. | Trace key information flows; identify which nodes receive what; map information barriers; check for gatekeeper filtering; test whether critical information reaches decision-makers. |
| IV Temporal | Cascade timescales are invisible. Bottleneck activation and queue dynamics are lagged and hard to detect in real-time. Temporal fragmentation means network connectivity appears stable snapshot-wise but is disconnected if you account for timing. | Apparent stability interrupted by sudden cascades; slow responses to urgent signals; oscillating behaviour in flow dynamics; temporal analysis showing fragmentation that is not visible in static maps. | Map cascade timescales and compare to decision/control response times. Test for phase lag in feedback. Analyse network rewiring over time. Check for edge-temporality (connections that only exist at certain times). |
| V  Relational | Hub-and-spoke topology distorts relationships; peripheral nodes have only mediated contact with each other, reducing trust and direct agency. Bridges are often in asymmetric relationships with both clusters. Bridge roles create hidden dependencies and resentments. | Brittle relationships (low trust, high formality); dependency on intermediaries that is not acknowledged; bridge occupants expressing burden or invisibility; clusters viewing each other through bridge's lens (which may distort). | Map direct vs mediated relationships. Interview bridges and cluster members separately. Test whether removing bridge breaks relationship or whether clusters can relate directly. Assess emotional tone (resentment, invisibility) at bridges. |
| VI Ontological | Different clusters may use different terminology and concepts to describe reality, making it difficult to translate across cluster boundaries. Bridges may be forced to choose which cluster's ontology to adopt, creating lossy translation. Network structure reflects and reinforces ontological boundaries. | Language and framing differ across clusters; bridges struggling to translate without losing meaning; disputes over definitions; communities "talking past" each other. | Collect key terms and definitions from each cluster; create a glossary comparing terms. Have bridges articulate translation challenges. Test whether concepts from cluster A are understood the same in cluster B. |

## 8.4 Embedded Hiddens Micro-Protocol (standard prompts)

1) Run the Tier 1 scan across all six Hiddens meta-categories (early pass; assume Hiddens exist) (OG §7.1).
2) Shortlist candidate Hiddens; for each, rate: mechanism, reducibility, detectability, agency, consequence (OG §7.2).
3) Assign at least one detection interface and one remediation interface per high-consequence Hidden; map interaction chains (OG §7.2).
4) Run the hiddens source analysis and record unresolved Unknowns/Hiddens.
5) If Tier 1 symptoms point to a specific hiding mechanism (e.g., structural hole blindness, echo chamber lock-in), invoke the appropriate Targeted Hiddens Discovery Scan (OG §7.5; select using Triage Matrix at §7.5.3).
6) After executing the framework protocol and any available tests/probes, re-run Tier 1 (pre-closure pass) and note deltas.
7) Produce Residual Unknowns (with mechanism + impact + next probe) and apply the Escalation Rule (§8.5) if any residual could change decisions, accountability, or remediation priorities. Produce closure artefacts per OG §7.4.

## 8.5 Escalation Rule (Tier 3 – full-spectrum Hiddens escalation)

Escalate to full-spectrum Hiddens taxonomy run with stricter evidence/provenance and stronger disconfirming tests if any apply:
- Hub dependency is material to critical outcomes (failure of hub would cascade system-wide).
- Bridge severance or bridge bottleneck is plausible in threat scenario (adversary, accident, or intentional).
- Cascade risk is high-consequence (safety, regulatory, existential, severe harm potential).
- Structural isolation or echo chambers are suspected to persist despite interventions.
- Positional blindness is preventing realistic coordination or decision-making.
- Adversarial context (competitive clusters, gatekeeping for advantage, deliberate topology distortion).
- Persistent coordination failures despite good intentions suggest hidden structural roots.
- Residual unknowns could change decisions on topology redesign, intervention timing, or risk allocation.

Tier 3 execution (per OG §D.6.10):
- Expands beyond Tier 1 categories into full Hiddens taxonomy (full-spectrum exploration including Hidden-02 Blind Spot, Hidden-06 Distortion, Hidden-09 Aggregation Blindness, Hidden-10 Model Blindness, Hidden-14 Fragmentation, Hidden-15 Latency, and others).
- Tightens controls: stronger provenance requirements for connectivity claims, more explicit competing hypotheses about cascade pathways, more rigorous disconfirming tests (e.g., "If this bridge is not critical, then this cluster should function independently; test: does it?").
- When adversarial intent is plausible: adds concealment vectors (deliberate hiding of topology, creation of shadow networks, fake redundancy) and independent corroboration paths (multiple sources, independent verification).
- May justify near-full group coverage (G1–G6) when warranted by routing decision tree branch A6 (OG §4.3), especially if topology redesign is being considered.

# 10. Framework Principles (mandatory)

## 10.1 Principles Table
| Principle name | Description |
|---|---|
| 1. Topology is Real and Consequential | Network structure is not decorative or merely descriptive; it shapes what flows where, who can see what, and how cascades propagate. Topology is often hidden but always consequential. Analysis must make topology explicit and ground decisions in topological facts and uncertainties. |
| 2. Position Matters More Than Good Intention | An agent in a well-positioned node accomplishes more with less effort than an isolated agent with superior competence. Conversely, an isolated node cannot succeed regardless of capacity. Interventions and accountability must account for positional constraints, not just personal agency. |
| 3. Structural Holes Are Power | Bridges that span structural holes have disproportionate leverage, visibility, and often invisible gatekeeping power. Bridges are often not recognised as powerful by themselves or by those they serve. Design and governance must account for bridge power and create transparency and redundancy at bridges. |
| 4. Cascades Are Topological, Not Random | Failure, information, or disturbance propagation is determined largely by network topology, not by randomness. Cascade pathways can be mapped, bottlenecks can be identified, and interventions can be designed to prevent or accelerate cascades. Surprises from cascades are evidence of prior topological blindness. |
| 5. Visibility Is Position-Dependent | Different network positions perceive different realities. Hub view is not "objective"; peripheral view is not "limited." Each position sees what its topology allows and cannot see what it is not connected to. Governance and decision-making must integrate multiple positions rather than privilege the hub view. |
| 6. Topology Changes Have Unintended Consequences | Removing a hub, adding a bridge, breaking a cycle, or rewiring clusters creates cascading effects throughout the network. Interventions must be tested against scenarios and monitored for second-order effects. Static topology analysis followed by abrupt change often produces worse outcomes than gradual rewiring with feedback. |

# 11. Glossary (local domain terms; recommended; mandatory for "Stable" status)

## 11.1 Local domain glossary
| Term | Definition |
|---|---|
| Betweenness centrality | Measure of how often a node lies on the shortest path between other nodes. High betweenness indicates a node is a bridge or connector. Nodes with high betweenness often act as gatekeepers or bottlenecks. |
| Bridge | A node (or edge) connecting two otherwise-disconnected clusters or groups. Bridges span structural holes and have disproportionate influence over cross-cluster flow. |
| Centrality | General measure of how central or important a node is in the network. Various measures exist: degree centrality (number of connections), betweenness centrality (how often on shortest path), closeness centrality (average distance to other nodes), eigenvector centrality (connected to important nodes). |
| Clustering coefficient | Measure of local clustering; probability that if A is connected to B and B is connected to C, then A is connected to C. High clustering indicates tight local groups. |
| Connected component | A subset of nodes such that every node is reachable from every other node by following edges. A network with multiple connected components is fragmented; nodes in different components cannot communicate directly. |
| Degree | Number of edges connected to a node. Nodes with high degree are well-connected; nodes with degree 0 are isolated; nodes with degree 1 are pendant nodes. |
| Density | Proportion of possible edges that are present in the network. Density = (actual edges) / (possible edges). Ranges from 0 (no edges) to 1 (fully connected). |
| Diameter | Maximum shortest-path length between any two nodes in the network. Large diameter indicates a sparse, elongated network; small diameter indicates good connectivity. |
| Echo chamber | Dense cluster with few outward bridges; information circulating within the cluster reinforces internal consensus; disconfirming information from outside does not arrive or is discounted. |
| Flow (or contagion) | Propagation of information, resources, disturbance, or failure through the network via edges and nodes. Flow speed depends on topology (fast through hubs, slow across sparse areas, blocked at severed bridges). |
| Gatekeeper | A node controlling access to or flow through a structural hole or bridge. Gatekeepers can filter, distort, or extract rents on cross-bridge flow. |
| Graph | Formal mathematical object representing a network: set of nodes and set of edges. Graph theory provides tools for network analysis. |
| Hub | A node with notably high degree or centrality. Hubs are well-connected and often central to network function. Hub failure can cascade system-wide. |
| Isolated node | A node with degree 0 (not connected to any other node) or degree 1 (connected only to one other node). Isolated nodes cannot access network-wide information or resources. |
| Latency | Time delay in propagation through the network. Long paths have high latency; hubs can reduce latency by shortcutting paths, but hubs also create queuing delays if saturated. |
| Modularity | Measure of network community structure; quantifies whether the network decomposes into clusters with high internal density and low between-cluster density. High modularity indicates strong community structure. |
| Network diameter | Length (in hops) of the longest shortest path between any two nodes. Indicator of network sparsity and connectivity. |
| Outdegree and indegree | For directed graphs: outdegree is number of outgoing edges; indegree is number of incoming edges. Distinction matters for directed flow (e.g., authority, communication direction). |
| Path length | Number of edges in a shortest path between two nodes. Average path length is a network-wide metric; long average path length indicates sparse, fragmented networks; short average path length indicates well-connected networks. |
| Reachability | Whether and how quickly one node can reach another. In fragmented networks, some node pairs have zero reachability (not connected at all). |
| Resilience | Robustness of network to node/edge removal. Resilient networks maintain connectivity and function despite failures; fragile networks degrade or collapse at single point failures. |
| Rewiring | Change in network structure over time; edges appear/disappear, nodes enter/exit, connection patterns shift. Dynamic networks require temporal analysis. |
| Scale-free network | Network where degree distribution follows a power law; few nodes have very high degree (hubs), many nodes have low degree. Common in real-world networks; resilient to random failure but vulnerable to targeted hub attack. |
| Shortest path | Path between two nodes with the minimum number of edges. Network analysis often focuses on shortest paths as the primary communication or influence routes. |
| Small-world network | Network combining high local clustering (like a lattice) with short average path length (like a random graph). Achieved through occasional long-range shortcuts; common in real networks; exhibits rapid global spread despite local clustering. |
| Structural hole | Gap (absence of edges) between clusters or groups. Bridges spanning structural holes have special power and visibility. |
| Topology | Pattern of connections in a network; the set of nodes and edges. Topology determines what flows where and how. Same actors in different topologies behave differently. |
| Weighted graph / edge weights | Edges may have weights (strength, capacity, latency, trust, etc.). Weighted networks are more realistic but more complex to analyse. |
